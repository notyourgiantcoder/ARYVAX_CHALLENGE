## Approach

The assignment is explicitly **not** a standard classification problem. Three real-world complications shaped every design decision:

**1. Noisy labels.**  
25.9% of training rows (311/1200) share a journal text with at least one other row that has a *different* label. Text like "it was fine" appears 12 times across 6 different emotional states. These are not mistakes — they reflect the core truth that short ambiguous text is genuinely unresolvable without context. The pipeline identifies all conflict texts at load time and uses them directly to drive the uncertainty layer.

**2. Weak metadata signal.**  
Despite labels being synthetically generated from metadata combinations, the ablation study shows metadata alone achieves only ~17% F1-macro. Text carries 82% of feature importance. This means the model correctly learned to rely on what the user actually said, not just their biometric readings.

**3. The task is guidance, not classification.**  
Predicting the emotional state is input to a decision, not the end goal. The system is designed around: understand → decide → guide. Prediction confidence feeds directly into whether a recommendation is delivered with certainty or flagged as uncertain.

## Feature Engineering

### Text features
- **TF-IDF** (3000 features, unigrams + bigrams, sublinear TF, Laplace smoothing via `smooth_idf=True`)
- Short/vague texts (≤3 words) are enriched before vectorization: `"ok"` becomes `"brief reflection forest morning session"` — preserving the ambience and time-of-day signal that the user implicitly provided by starting the session.

### Metadata features
Columns are auto-detected from the CSV by type (`float()` probe). No column names are hardcoded in the pipeline.

| Feature | Type | Note |
|---|---|---|
| `energy_level`, `stress_level`, `sleep_hours`, `duration_min` | numeric | Median-imputed from training data |
| `face_emotion_hint`, `time_of_day`, `previous_day_mood`, `ambience_type`, `reflection_quality` | ordinal | Semantically ordered (calm→aroused axis, temporal progression) |
| `stress_energy_ratio` | derived | `stress / max(energy, 0.1)` — captures exhausted-but-stressed pattern |
| `is_night` | derived | Binary flag; night sessions have different action profiles |
| `is_short_vague` | derived | Flags ≤3-word texts so the model learns to down-weight text confidence |

**Missing values:** `SimpleImputer(strategy="median")` fitted on training data. Medians are computed, not assumed. Categorical unknowns (missing `face_emotion_hint`, `previous_day_mood`) are encoded as a dedicated "unknown" category — no values invented.

**Semantic ordering** for categorical features is defined in `config.json` (generated by `generate_config.py`) and intentional — `OrdinalEncoder` with `calm_face=0, tense_face=3` encodes the arousal axis, which alphabetical sorting would destroy.

## Model Choice

### Emotional state (Part 1)
**GradientBoostingClassifier** wrapped in **CalibratedClassifierCV** (isotonic regression).

- GBM handles the mixed sparse (TF-IDF) + dense (metadata) feature matrix well without needing separate scaling
- Calibration is non-optional here: raw GBM probabilities are overconfident, which would make the uncertainty layer meaningless. Isotonic calibration ensures a 0.6 confidence prediction is actually correct ~60% of the time
- Alternative considered: LogisticRegression — faster and more interpretable, but GBM handles the interaction between text and metadata signals better

### Intensity (Part 2)
**GradientBoostingClassifier** + **CalibratedClassifierCV**, evaluated with **MAE** not accuracy.

Intensity is framed as **ordinal classification**, not regression. Predicting 2.7 is meaningless on a 1–5 scale. But treating it as plain multiclass ignores the ordering — predicting 5 when the truth is 1 should cost more than predicting 2. MAE captures this; the pipeline reports both accuracy (for comparability) and MAE (as the primary metric).

### Why not fine-tuned BERT / DistilBERT?
With 1,200 training samples and 25.9% label noise, fine-tuning a 66M-parameter model would overfit badly. GBM with TF-IDF features is more appropriate at this data scale and is fully explainable in an interview — you can point to specific tokens and importances.

## How to Run

```bash
# Full pipeline (train + predict + all analysis)
python arvyax_pipeline.py

# Regenerate config if data schema changes
python generate_config.py

# Inference only (load saved model, predict on new CSV)
python - << 'EOF'
import joblib, csv
from arvyax_pipeline import load_csv, build_X, decision, supportive_msg, get_uncertainty
import numpy as np

bundle   = joblib.load("models.joblib")
test     = load_csv("new_test.csv")
X, _, _, _, _, _ = build_X(test,
    bundle["num_cols"], bundle["cat_cols"],
    tfidf=bundle["tfidf"], imputer=bundle["imputer"], enc=bundle["enc"])

pred_s = bundle["le"].inverse_transform(bundle["clf_s"].predict(X))
pred_i = bundle["clf_i"].predict(X)
proba  = bundle["clf_s"].predict_proba(X)
unc    = get_uncertainty(proba, test, bundle["conflict_set"])

for i, r in enumerate(test):
    what, when = decision(pred_s[i], pred_i[i],
                          r.get("stress_level"), r.get("energy_level"),
                          r.get("time_of_day","afternoon"))
    conf, flag = unc[i]
    print(r["id"], pred_s[i], int(pred_i[i]), conf, flag, what, when)
EOF
```
